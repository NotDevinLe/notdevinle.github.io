previously, for my system I was hashing an entire prompt and checking for it across all my tpus. However, this is not necessarily a good idea because:
    - this doesn't take into account if certain kv pairs got evicted from a worker's memory
    - we want to look at smaller prefixes

the solution is to use a useful feature of vLLM's engine `kv_events` which is a parameter you can enable to instantiate a zmq publisher in which you can now spin up your own zmq subscriber and keep track of what is truly in a worker's kv cache.

## architecture

lets start with how vllm interacts with our redis database.

```bash
[Live Path: Fast/Lossy]
vLLM Worker (PUB :5557) ----> rproxy (SUB) ----> Redis Index

[Recovery Path: Slow/Guaranteed]
vLLM Worker (ROUTER :5558) <--> rproxy (DEALER) ----> Redis Index
```

this is the absolute barebones of what's happening which is quite simple. the worker will publish everything on the port 5557 and there is a publisher-subscriber relation between the two of them. 

each request comes with an incementing id tag that helps us know what the last successful request we got from the publisher was. If we see that there is a gap (ie. last id we had was 100 but we just received 105) then we need a way to get this data back reliably. This is why we add a dealer such that we introduce a router-dealer relationship. This is guaranteed to be correct but is much slower and is called whenever a subscriber sees its missing information.
